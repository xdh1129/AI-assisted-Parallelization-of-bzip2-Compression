‚óŹ Confirmed from the code. Here's the breakdown.

  They produce the same output ‚ÄĒ so the difference is purely how, not what

  Both emit identical bzip2-format streams (that's why the compression ratio is
  the same, 0.2607 vs 0.2608). A .bz2 block is built in stages:

  input ‚Üí RLE1 ‚Üí BWT (sort) ‚Üí MTF ‚Üí RLE2 ‚Üí Huffman ‚Üí bits

  pbzx and lbzip2 do the MTF / RLE / Huffman stages essentially the same way.
  The entire speed difference lives in one stage: the BWT.

  The BWT is the expensive part, and it's a sorting problem

  The Burrows‚ÄďWheeler Transform reorders the (up to) 900,000 bytes of a block by
  sorting all the suffixes of the block. The sorted order is unique, so both
  tools get the same BWT ‚ÄĒ but suffix-sorting is ~47% of the runtime, and that's
  where the algorithms diverge.

  pbzx ‚Üí bzip2's blockSort (Seward's algorithm) ‚ÄĒ confirmed: bz_block.c includes
  <bzlib.h> and calls BZ2_bzCompress, which lands in BZ2_blockSort ‚Üí mainSort
  (third_party/bzip2/blocksort.c). It buckets suffixes by their first 2 bytes,
  then quicksorts each bucket by comparing suffixes byte-by-byte. It has a
  "quadrant" trick to reuse some comparison work and a work-budget: on nasty
  repetitive data it gives up and falls back to the slower fallbackSort. The
  weakness is those repeated byte-by-byte string comparisons ‚ÄĒ on data with long
  repeats it does a lot of redundant work.

  lbzip2 ‚Üí divbwt = libdivsufsort (Yuta Mori) ‚ÄĒ confirmed: lbzip2/src/divbwt.c
  is a full suffix-array constructor. Instead of comparison-sorting everything,
  it uses induced sorting:
  1. classify suffixes into types, then sort only the smaller "type B*" subset
  with sssort (a multikey introsort over short substrings ‚ÄĒ these were the hot
  ss_mintrosort / ss_swapmerge functions in the profile),
  2. induce the order of all the other suffixes from that sorted subset with
  trsort (the tr_partition / tr_introsort functions), which handles
  periodic/repetitive runs cheaply.

  That's the "optimized algorithm" you noticed. divsufsort does far less
  redundant comparison work, scales closer to O(n log n) robustly, and never
  needs a slow fallback ‚ÄĒ so it computes the same BWT roughly 1.66√ó faster 
  single-threaded.

  Secondary difference: the parallelization architecture (not where the per-core
  win comes from)

  - pbzx: OpenMP #pragma omp parallel for over independent 900 KB blocks ‚ÄĒ
  simple fork/join, each thread compresses whole blocks via libbz2. (You also
  added a per-thread workspace arena to avoid malloc churn.)
  - lbzip2: a hand-built pthread pipeline ‚ÄĒ one reader thread, a worker pool,
  one writer/muxer thread, linked by priority queues that keep output in order
  and overlap I/O with compute.

  Both parallelize at the block level, so this architecture difference mostly
  affects scaling smoothness and I/O overlap ‚ÄĒ not the raw per-core throughput.
  The per-core speedup is almost entirely the BWT: blocksort vs divsufsort.

  One-line summary: same compression, same bzip2 format ‚ÄĒ lbzip2 just swaps 
  bzip2's comparison-based suffix sort for a suffix-array (induced-sorting) BWT 
  that does the identical job with less work.
